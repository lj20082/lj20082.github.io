Objective-C类的 +load 和 +initialize方法

发表于 2018-06-18 | | 阅读次数:
字数统计: 2,522 | 阅读时长 ≈ 12

Objective-C中的绝大多数类都继承于NSObject类，而在NSObject类中有2个非常特殊的方法+load和+initialize，用于类的初始化。这2个看似非常简单的类方法在许多方面让人困惑，比如：

  1. 子类、父类、分类中的相应方法什么时候会被调用
  2. 是否需要在子类的实现中显式的调用父类的实现
  3. 每个方法到底会被调用多少次

那么我们结合runtime源码（objc4-723）看看其内部具体实现。

+load

+load方法是当类或分类被添加到Objective-C runtime时被调用的，实现这个方法可以让我们在类加载的时候执行一些类相关的行为。子类的+load方法会在它的所有父类的+load方法之后执行，而分类的+load方法会在它主类的+load方法之后执行。

这个函数的作用就是提前准备好满足+load方法调用条件的类和类别，以供接下来的调用，其中，在处理类时，调用了同文件中的static void schedule_class_load(Class cls)方法来这些具体的操作。

其中，函数第9行代码对入参的父类进行了递归调用，以确保父类优先的顺序。void prepare_load_methods(const headerType *mhdr)函数执行完毕后，当前所有满足+load调用条件的类和分类就分别存放在全局变量static struct loadable_class *loadable_classes 和static struct loadable_category *loadable_categories中了。

准备好类和分类后，接下来就是对它们的+load方法进行调用了。打开文件objc-loadmethod.mm，找到其中的void call_load_methods(void)函数。

然后，循环调用所有类的+load方法，这里是直接使用的函数内存地址的方法(*load_method)(cls, SEL_load),而不是使用发送消息objc_msgSend的方式。

这样的调用方式使得+load方法拥有了一个非常有趣的特性，那就是子类、父类和分类中的+load方法是被区别对待的，也就是说子类没有实现+load方法，那么当它被加载时runtime是不会去调用父类的+load方法的。同理如果一个类和它的分类都实现了+load方法时，2个方法都会被调用。因此，我们可以利用这个特性做一些特别的事情，比如方法混淆Method Swizzling

+initialize

+initialize方法是在类或它的子类收到第一条消息前被调用的，这里所值的消息包括实例方法和类方法的调用。也就是说+initialize方法是以懒加载的方式被调用的，如果程序一直没有给某个类或它的子类发送消息，那么这个类的initialize方法永远也不会被调用，这样设计的好处是节省系统资源，避免不必要的开销

总结

通过阅读 runtime 的源码，我们知道了 +load 和 +initialize 方法实现的细节，明白了它们的调用机制和各自的特点。下面我们绘制一张表格，以更加直观的方式来巩固我们对它们的理解：

+load +initialize
调用时机 被添加到 runtime 时 收到第一条消息前，可能永远不调用
调用顺序 父类->子类->分类 父类->子类
调用次数 1次 多次
是否需要显式调用父类实现 否 否
是否沿用父类的实现 否 是
分类中的实现 类和分类都执行 覆盖类中的方法，只执行分类的实现
